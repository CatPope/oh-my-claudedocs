# Purpose

프로젝트의 모든 문서와 코드를 대조 검토하여 맥락 일관성, 누락 항목, 코드-문서 괴리를 점수로 평가하고 `docs/dev/review/` 디렉토리에 보고서를 생성한다. 보고서 작성 후 발견된 문제를 자동 수정한다.

# Use When

- 개발 단계 전환 시 (기획→설계, 설계→구현 등)
- 주요 결정 변경 후
- 모듈 또는 기능 단위 작업 완료 후
- 정기 검토 시
- 사용자가 `/doc-review`를 실행할 때

# Do Not Use When

- 문서가 아직 작성되지 않았을 때
- 단순 오타 수정 시

# Arguments

| 인자 | 설명 |
|------|------|
| `full` | 전체 심층 검토 (기본값) |
| `quick` | 빠른 요약 검토 (존재/비존재, 주요 불일치만) |
| 모듈명 | 특정 모듈/영역만 검토 (예: `auth`, `api`, `frontend`) |

# 실행 빈도 정책

프로젝트 규모에 따라 실행 빈도가 달라진다:

| 규모 | 단위 | 실행 시점 | 트리거 |
|------|------|-----------|--------|
| **대규모** (10+ 모듈) | 모듈 단위 | 각 모듈 작업 완료 시 | dev-team이 서브 에이전트로 자동 위임 |
| **중규모** (3~9 모듈) | 기능 단위 | 주요 기능 완료 시 | dev-team이 서브 에이전트로 자동 위임 |
| **소규모** (1~2 모듈) | 프로젝트 단위 | 단계 전환 시 | dev-team이 서브 에이전트로 자동 위임 |

dev-team 오케스트레이터는 프로젝트 규모를 파악한 후, 위 정책에 따라 적절한 시점에 `/doc-review`를 서브 에이전트에 위임한다.

# Steps

## 0. 검사 난이도 질문

사용자에게 검토 깊이를 확인한다:

> 문서 검토 난이도를 선택해주세요:
> 1. **경량** — 구조/존재 여부 + 주요 불일치만 (5분 이내)
> 2. **표준** — 문서 간 일관성 + 코드-문서 정합성 (기본값)
> 3. **심층** — 전체 코드 라인 대조 + ADR 추적 + 미래 방향성 분석

`quick` 인자 시 자동으로 경량, `full` 인자 시 자동으로 심층 선택.

## 1. 문서 인벤토리

1. `docs/dev/` 디렉토리 스캔
2. 존재하는 문서 목록 작성
3. 누락된 필수 문서 식별 (현재 개발 단계 기준)

## 2. 개별 문서 검증

각 문서에 대해 (목차만 먼저 읽고 필요한 섹션만 접근):

- **구조 검증**: 15줄 헤더 + 목차 L값 구조 준수 여부
- **완성도**: 빈 섹션, TODO, placeholder 식별
- **최신성**: 마지막 수정일 vs 관련 코드 변경일 비교 (`git log`)

## 3. 문서 간 일관성 검토

- **용어 일관성**: 동일 개념에 다른 용어 사용 감지
- **결정 정합성**: SRS/PRD 요구사항 ↔ Architecture 설계 ↔ DetailedSpec 구현 명세
- **기술 스택 일관성**: CLAUDE.md ↔ Architecture ↔ 실제 package.json/코드

## 4. 코드-문서 괴리 분석

- ADR에 기록된 결정 vs 실제 코드 구현 비교
- SRS/PRD 요구사항 vs 실제 구현된 기능 비교
- Architecture 설계 vs 실제 코드 구조 비교
- 문서에 없는 결정이 코드에 반영된 경우 식별
- test-plan 테스트 범위 vs 실제 테스트 파일 비교
- deploy-guide 배포 절차 vs 실제 CI/CD 설정 비교

## 5. 미래 방향성 검토 (심층만)

- SRS/PRD에 정의되었으나 미구현된 요구사항 목록
- Architecture에서 계획되었으나 미구현된 컴포넌트 목록
- 기술 부채 식별 (임시 해결, TODO, 제한사항)

## 6. 점수 산출

각 영역별 100점 만점으로 채점한다:

| 영역 | 배점 | 감점 기준 |
|------|------|-----------|
| **구조 준수** | 100 | 15줄 헤더 위반(-20), L값 불일치(-10), 목차 누락(-15) |
| **문서 완성도** | 100 | 빈 섹션(-15/개), TODO 잔존(-5/개), placeholder(-10/개) |
| **문서 간 일관성** | 100 | 용어 불일치(-10/건), 결정 충돌(-20/건), 스택 불일치(-15/건) |
| **코드-문서 정합성** | 100 | 미반영 요구사항(-15/건), 미기록 결정(-10/건), 구조 불일치(-20/건) |
| **최신성** | 100 | 코드 변경 후 미갱신(-15/문서), 30일+ 미수정(-10/문서) |
| **종합** | 가중 평균 | 구조(10%) + 완성도(25%) + 일관성(25%) + 정합성(30%) + 최신성(10%) |

### 등급 기준

| 등급 | 점수 | 의미 |
|------|------|------|
| **A** | 90~100 | 우수 — 즉시 배포 가능 |
| **B** | 75~89 | 양호 — 경미한 보완 필요 |
| **C** | 60~74 | 보통 — 보완 후 진행 |
| **D** | 40~59 | 미흡 — 상당한 수정 필요 |
| **F** | 0~39 | 부족 — 재작성 권장 |

## 7. 보고서 작성

결과를 `docs/dev/review/review-YYYY-MM-DD.md`에 저장한다. 동일 날짜에 이미 파일이 있으면 타임스탬프를 추가한다.

### 보고서 형식

```markdown
# 문서 검토 보고서

- **검토 일시**: YYYY-MM-DD HH:mm
- **검토 난이도**: 경량 / 표준 / 심층
- **검토 범위**: 전체 / 모듈명
- **종합 등급**: A~F (XX점/100점)

## 1. 점수 요약

| 영역 | 점수 | 등급 | 주요 이슈 |
|------|------|------|-----------|
| 구조 준수 | /100 | | |
| 문서 완성도 | /100 | | |
| 문서 간 일관성 | /100 | | |
| 코드-문서 정합성 | /100 | | |
| 최신성 | /100 | | |
| **종합** | **/100** | | |

## 2. 문서 인벤토리

| 문서 | 상태 | 최종 수정 | 비고 |
|------|------|-----------|------|

## 3. 발견 이슈 (심각도순)

### 높음
- [코드-문서 정합성] 이슈 설명

### 중간
- [일관성] 이슈 설명

### 낮음
- [최신성] 이슈 설명

## 4. 코드-문서 괴리 상세

| # | 문서 | 문서 내용 | 코드 실제 | 괴리 유형 |
|---|------|-----------|-----------|-----------|

## 5. 권장 조치

- [ ] [높음] 조치 항목
- [ ] [중간] 조치 항목
- [ ] [낮음] 조치 항목

## 6. 이전 검토 대비 변화 (해당 시)

| 영역 | 이전 점수 | 현재 점수 | 변화 |
|------|-----------|-----------|------|
```

## 8. 자동 수정

보고서 작성 완료 후, 발견된 문제를 자동 수정한다:

### 수정 우선순위

1. **구조 위반** — 15줄 헤더, L값 목차 자동 교정
2. **심각도 높음** — 코드-문서 괴리, 결정 충돌 수정
3. **심각도 중간** — 용어 통일, 누락 섹션 보완
4. **심각도 낮음** — 최신성 갱신, placeholder 제거

### 수정 방식

- 각 수정은 **서브 에이전트(writer)**에 위임한다
- 독립적인 문서 수정은 병렬로 진행한다
- 수정 후 보고서의 "권장 조치" 체크리스트를 갱신한다
- 자동 수정이 불가능한 항목(사용자 판단 필요)은 체크리스트에 남긴다

### 수정 후 재채점

자동 수정 완료 후 변경된 영역만 재채점하여 보고서 하단에 추가한다:

```markdown
## 수정 후 재채점

| 영역 | 수정 전 | 수정 후 | 변화 |
|------|---------|---------|------|
| **종합** | XX점 | YY점 | +ZZ점 |
```

## 9. 이전 검토와 비교

`docs/dev/review/` 내 이전 보고서가 존재하면 점수 변화를 추적하여 "이전 검토 대비 변화" 섹션을 채운다.

# dev-team 연동

dev-team 오케스트레이터가 프로젝트 규모에 따라 자동 위임하는 방법:

```
메인 (오케스트레이터)
  │
  ├─ [대규모] 모듈 A 구현 완료 → Agent(doc-reviewer) "module auth"
  ├─ [대규모] 모듈 B 구현 완료 → Agent(doc-reviewer) "module api"
  │
  ├─ [중규모] 기능 X 완료 → Agent(doc-reviewer) "module feature-x"
  │
  └─ [소규모] 단계 전환 시 → Agent(doc-reviewer) "full"
```

에이전트 위임 시 프롬프트 예시:
```
docs/dev/review/ 디렉토리에 검토 보고서를 작성하라.
검토 범위: [전체 | 모듈명]
검토 난이도: [경량 | 표준 | 심층]
보고서 작성 후 심각도 높음/중간 이슈를 자동 수정하라.
수정 후 재채점 결과를 보고서에 추가하라.
```
