# Purpose

프로젝트의 모든 작성된 문서를 검토하여 일관성, 결정 탈선, 현재 개발 흐름과의 정합성, 미래 방향성을 점검한다.

# Use When

- 개발 단계 전환 시 (기획→설계, 설계→구현 등)
- 주요 결정 변경 후
- 정기 검토 시
- 사용자가 `/doc-review`를 실행할 때

# Do Not Use When

- 문서가 아직 작성되지 않았을 때
- 단순 오타 수정 시

# Arguments

| 인자 | 설명 |
|------|------|
| `full` | 전체 심층 검토 (기본값) |
| `quick` | 빠른 요약 검토 (존재/비존재, 주요 불일치만) |
| 섹션명 | 특정 문서만 검토 (예: `architecture`, `srs`) |

# Steps

## 0. 문서 인벤토리

1. `docs/dev/` 디렉토리 스캔
2. 존재하는 문서 목록 작성
3. 누락된 필수 문서 식별 (현재 개발 단계 기준)

## 1. 개별 문서 검증

각 문서에 대해 (목차만 먼저 읽고 필요한 섹션만 접근):

- **구조 검증**: 15줄 헤더 + 목차 구조 준수 여부
- **완성도**: 빈 섹션, TODO, placeholder 식별
- **최신성**: 마지막 수정일 vs 관련 코드 변경일 비교 (`git log`)

## 2. 문서 간 일관성 검토

- **용어 일관성**: 동일 개념에 다른 용어 사용 감지
- **결정 정합성**: SRS/PRD의 요구사항 ↔ Architecture의 설계 ↔ DetailedSpec의 구현 명세 일치 여부
- **기술 스택 일관성**: CLAUDE.md ↔ Architecture ↔ 실제 package.json/코드

## 3. 결정 탈선 감지

- ADR에 기록된 결정 vs 실제 코드 구현 비교
- SRS/PRD의 요구사항 vs 실제 구현된 기능 비교
- Architecture의 설계 vs 실제 코드 구조 비교
- 문서에 없는 결정이 코드에 반영된 경우 식별

## 4. 개발 흐름 정합성

- 현재 Git 브랜치/커밋 히스토리와 문서 단계의 일치 여부
- test-plan의 테스트 범위 vs 실제 테스트 파일 비교
- deploy-guide의 배포 절차 vs 실제 CI/CD 설정 비교

## 5. 미래 방향성 검토

- SRS/PRD에 정의되었으나 미구현된 요구사항 목록
- Architecture에서 계획되었으나 미구현된 컴포넌트 목록
- 기술 부채 식별 (임시 해결, TODO, 제한사항)

## 6. 검토 보고서 출력

검토 결과를 구조화된 형식으로 출력:

### 보고서 구조

```
## 문서 검토 보고서

### 1. 문서 인벤토리
| 문서 | 상태 | 최종 수정 | 비고 |
|------|------|-----------|------|

### 2. 일관성 이슈
- [심각도: 높음/중간/낮음] 이슈 설명

### 3. 결정 탈선
- [문서] 결정 내용 → [코드] 실제 구현 → 차이점

### 4. 개발 흐름
- 현재 단계: [기획/설계/구현/테스트/완료]
- 단계 정합성: [일치/불일치]

### 5. 미구현 항목
- [우선순위] 항목 설명

### 6. 권장 조치
- [ ] 조치 항목 1
- [ ] 조치 항목 2
```

`quick` 모드에서는 1, 6만 출력. `full` 모드에서는 전체 출력.
